OVO-Bench 的核心 motivation 可以拆成三层。
第一层，传统视频 QA 太像“考后阅读理解”。模型通常可以看到完整视频，然后回答“视频里发生了什么”。但真实 online assistant 不能偷看未来。比如 AR 眼镜场景里，用户在第 30 秒问“现在桌上有什么”，模型就只能基于第 30 秒之前和附近的画面回答，不能用第 60 秒的信息。
第二层，online video understanding 不只是“实时识别当前画面”。你前面问过为什么它要分这么多任务，其实作者想强调：用户的问题可能指向三种时间位置。答案可能在过去，比如“我刚才把筛子放哪了”；可能在当前，比如“现在洗衣机上有什么”；也可能在未来，比如“等他打开盒子后告诉我里面是什么”。所以 online benchmark 不能只测当前帧识别，还要测回忆过去和等待未来。
第三层，当前模型缺少 temporal awareness。它们可能识别能力不差，但不知道“用户问的是现在这个时间点”，所以容易把过去相似画面的内容拿来回答；也容易在当前证据不足时硬答，产生 hallucination。OVO-Bench 就是为了把这些失败暴露出来。论文和项目页都把 temporal awareness 作为 offline 与 online video LLM 的关键区别，并强调模型需要沿时间线寻找视觉线索。
OVO-Bench 的 method 不是提出一个新模型，而是提出一套 benchmark construction + evaluation pipeline。
第一步是 任务 taxonomy。作者把 online video understanding 分成三类：Backward Tracing：从过去找证据。Real-Time Visual Perception：从当前画面找证据。Forward Active Responding：当前证据不够，等待未来线索。
第二步是 数据构造。作者从已有人工标注数据集和网络爬取视频中收集 raw videos。这样既有标注质量，也有场景多样性。然后他们生成 meta-annotations，这里的 meta-annotation 不是普通 QA，而是带有精确时间戳的事件标注，因为 online evaluation 必须知道“线索到底在什么时候出现”。这一步用 VLLM 辅助生成，再人工 refine。最后生成多选题，并用视觉相关的干扰项提高难度。
第三步是 Chain-of-Time evaluation。这就是图 3 下半部分。用户问题来了以后，模型要判断：当前画面能不能回答？如果能，就是 Real-Time Visual Perception；如果不能，过去记忆能不能回答？如果能，就是 Backward Tracing；如果过去和当前都不够，就应该等待未来输入，也就是 Forward Active Responding。项目页也明确说 OVO-Bench 评测模型从 ongoing input 中寻找 temporal visual clues，并允许模型等待足够证据后再响应。

OVO-Bench: How Far is Your Video-LLMs from Real-World Online VideoUnderstanding?time-aware online interaction
motivation ：现有很多视频理解 benchmark 本质上还是 offline video QA，也就是模型看完整段视频后再回答问题；但真实 AR/VR、机器人或在线视频助手不是这样工作的，用户会在视频流进行中的某个时间点突然提问，模型只能根据当时已经看到的信息来判断答案，而且有时候不能立刻回答，而是要等未来线索出现后再回答。因此，作者认为真实 online video understanding 的关键不是单纯“看懂视频”，而是 temporal awareness：模型要知道问题是在什么时候被问的，并判断证据来自过去、当前，还是未来。OVO-Bench 把任务设计成三类：Backward Tracing 测模型能不能回忆过去事件，Real-Time Visual Perception 测模型能不能理解当前画面，Forward Active Responding 测模型能不能在当前证据不足时先等待、等未来线索出现后再回答。方法上，作者先从已有人工标注数据集和网络视频中收集原始视频，再用 GPT-4o、Gemini-1.5-Pro 等 VLLM 辅助生成带精确时间戳的 meta-annotations，并经过人工 refine，最后构造带视觉干扰项的多选题；评测时则沿着视频时间线 query 模型，模拟真实在线视频交互。OVO-Bench 更关注“时间意识”和 online response timing，而 EgoEverything 更关注 AR 第一视角中由 human attention / gaze 驱动的长期个人记忆问题。现在视频理解需要很多 benchmark因为“视频理解”已经被拆成了不同能力：长视频记忆、第一视角理解、当前画面定位、未来线索触发、注意力对齐、幻觉检测等，每个 benchmark 都在抓模型不同的失败面。

Streaming VideoLLMs for Real-time Procedural Video Understanding在做咖啡、做饭、组装这类长流程任务时，模型不能等视频结束后再理解，而是要边看视频边回答，比如“我现在在做什么”“下一步该做什么”“最终目标是什么”。长期历史用文字记忆，短期当前用视觉 token，二者交错放进同一个 cache，让模型可以实时理解和预测。
[NOTES]
现有 VideoLLM 多数适合 offline video understanding，但 AR assistant 这类真实场景需要模型实时理解长时间 procedural video。直接保存所有历史视觉 token 会导致 token 数、显存和计算量爆炸；只看短期画面又会丢失历史步骤信息。因此，作者想设计一种既能记住长期流程，又能保留当前细节，同时还能实时运行的 streaming VideoLLM
论文逻辑链
1. 真实需求：AR/实时助手需要边看边理解。比如用户正在用 Moka Pot 做咖啡，模型不能等整个视频结束后再回答，而是要实时判断：
“用户现在做到哪一步？”“之前已经完成了什么？”“下一步应该做什么？”
所以问题不是普通 offline video QA，而是 streaming procedural video understanding。
2. 核心困难：procedural video 又长、又有步骤依赖。做饭、组装、修东西这种视频不是单个动作识别，而是一串流程。
当前动作是否正确，下一步应该做什么，都依赖历史步骤。比如如果已经加水了，下一步可能是放滤网；如果已经放滤网了，下一步才是加咖啡粉。
所以模型需要记住长期历史。
3. 直接存全部视觉 token 不现实。如果每一帧都转成 visual tokens 存进 LLM context，视频越长，token 越多，显存和 attention 计算都会爆。
这也是普通 VideoLLM 难以直接用于实时长视频的原因。
4. 作者的关键观察：长期历史和短期当前不应该用同一种表示。
长期历史主要回答的是：
“之前做过哪些步骤？”
这个可以用文字表示，比如：
<L> add water to bottom<L> place filter onto bottom
不需要继续保存大量视觉帧。
但短期当前主要回答的是：
“现在手在做什么？物体有没有被拿起？有没有倒进去？”
这需要细粒度视觉信息，所以最近几秒仍然要保留 visual tokens。
所以作者提出一个很直观的思想：
过去发生了什么，用语言记；现在正在发生什么，用视觉看。
5. 但是只把视觉变成文字还不够，还要解决 cache 更新问题。
如果 long-term text cache 和 short-term visual cache 分开存，每次旧视觉 token 转成文字 token 时，可能需要重新计算 attention，导致 streaming 延迟不稳定。
所以作者提出 multimodal interleaved cache：
把 long-term text tokens 和 short-term visual tokens 放进同一个 causal cache 里。
这样模型可以持续增量更新，不需要大规模重算。
6. 另一个问题：短期 visual tokens 要关注哪里？
procedural video 的关键通常不是背景，而是 手和被操作物体。
比如做咖啡时，真正重要的是手有没有拿滤网、咖啡粉有没有倒进去，而不是桌子、墙面、厨房背景。
所以作者提出 DETR-QFormer：
它用 visual queries、hand queries、object queries 从 DINOv2 patch tokens 里提取信息，并用 hand-object bbox pseudo labels 训练，让视觉 token 更关注手和物体交互区域。

如果全存视觉 token，长视频 token 数会爆；如果先 verbalize 再分开存 long-term 和 short-term，会带来 attention recomputation；把它们放在同一条 token 序列里：<L><t1>, <L><t2>, <v6>, <v7>, <v8>, <v9> ...当 <v6> 这段旧视觉信息过期时，就把它 verbalize 成 <L><t3>，然后插入到同一个 interleaved cache 里。cache 始终是一个统一的 multimodal sequence，LLM 可以继续用 causal KV cache 增量更新，不需要大规模重算。作者认为短期视觉 token 不能随便压缩。因为 procedural video 的关键动作通常发生在 hand-object interaction 上，比如拿滤网、倒咖啡粉、压平表面。如果普通 Q-Former 把背景也压进去，token 就浪费了。所以 DETR-QFormer 加了 hand queries 和 object queries，并用 pseudo bbox 做辅助训练，让视觉 token 更集中在手和接触物体上。
[NOTES]
这篇论文的方法是 ProVideLLM，主要有三部分：
第一，long-term verbalization。把长期过去的视频片段转成文字步骤 token，比如 <L> add water to bottom，用语言压缩历史，减少视觉 token 数。
第二，short-term visual tokenization。最近几秒的视频仍然保留 visual tokens，用 DINOv2 + DETR-QFormer 提取 hand-object-focused visual tokens，让模型看清当前手和物体的细节。
第三，multimodal interleaved cache。把长期文字 token 和短期视觉 token 交错放进同一个 causal cache 里，让 LLM 可以边接收新视频边增量推理，避免每次 verbalization 都重新计算 attention。

LiveStar: Live Streaming Assistant for Real-World Online Video Understanding NeurIPS 2025motivation： 真实在线公开视频视频一帧一帧进来，模型要边看边决定 什么时候说、什么时候沉默、说什么。现有 online Video-LLM 很多靠学 <EOS> 表示“不说话”，但大部分帧都不需要输出，导致模型训练时大量学习“画面 → EOS”，容易破坏原本的视觉-语言对齐，还会让相邻相似帧出现一帧输出 narration、一帧输出 EOS 的不稳定监督。核心目标： 处理连续 frame-by-frame 输入，同时判断最优响应时机，并保持叙事连贯性。Method： 第一，训练时用 SCAM，也就是 Streaming Causal Attention Masks，把视频帧和 caption 交错组织起来，让模型学“当前视频前缀应该对应什么 narration”，而不是疯狂学 EOS；第二，推理时用 SVeD，也就是 Streaming Verification Decoding，每来一帧先检查上一句 narration 在当前画面下还合不合理，如果 PPL 变高超过阈值，就说明语义变了，需要生成新 narration；如果还合理，就保持沉默；第三，为了处理 10 分钟以上长视频，用 Peak-End Memory Compression 保留关键帧和每段最后总结（删除不重要的旧帧），再配合 streaming KV cache 加速（缓存历史KV）。论文还提出 OmniStar 数据集，覆盖 15 个真实场景和 5 个 online video understanding 任务（评测模型是否能在视频一帧一帧输入时，自己决定什么时候输出、什么时候沉默，并且输出内容是否准确、是否重复、是否和历史上下文连贯。）
最上面是实时视频流，模型不是看完整视频再回答，而是一帧一帧接收输入，并且只在语义变化时输出 narration。左边是旧方法，它训练时让大量非关键帧输出 EOS，但这样会导致 response-silence 极度不平衡，而且相邻相似帧可能一帧输出 narration、一帧输出 EOS，破坏视觉-语言对齐。中间是 LiveStar 的做法：训练时用 SCAM，把 frame 和 caption 交错起来，并 mask 掉容易复制的重复 caption，让模型继续学视频内容和语言的对应关系；推理时用 SVeD，每来一帧先检查上一句 narration 还是否适合当前画面，如果适合就沉默，如果不适合就生成新的 narration。右边最终展示的是一段连贯的 streaming narration，而不是每帧都重复输出。
设计指标看什么时候该重新decode，不保留重复语义以免误导模型&增加不必要kv cache

(a) Streaming Narration 表示任务场景。
视频是一帧一帧流入的，模型收到 prompt：Generate streaming narration in real time，重点不是每一帧都说话。时间轴上有些位置是 Narration，有些位置是捂嘴表情，表示 silent。这说明在线模型真正要做两件事：第一，理解当前视频内容。第二，判断现在该不该输出。
比如小猫一直在草地上跑，连续几帧语义差不多，就不需要每帧都重复说；等小猫从户外切到室内，语义变化了，才应该输出新的 narration。
(b) Existing Methods 讲的是旧方法。
训练阶段，旧方法把每一帧都变成一个输出目标：有些关键帧输出 Narration ，大量普通帧输出 <EOS>，表示 silence
问题是，这种方法太依赖 EOS。因为大部分时间都不该说话，所以训练数据里 EOS 特别多。结果模型不是在稳定学习“视频内容到语言描述”的对应关系，而是在学“很多画面都输出 EOS”。
更糟的是，相邻帧可能视觉很像，但监督完全不同：一帧要求输出 narration，下一帧要求输出 EOS。这样模型会很困惑，训练也不稳定。
推理阶段，旧方法每来一个 current frame，就 decode 一次，然后决定输出 <EOS> 还是 narration。这个方式既慢，又容易出现错误 timing：要么说太多，要么该说的时候不说。
(c) LiveStar (Ours) 是这篇文章的核心。
它分成训练和推理两部分。训练时，LiveStar 不再简单让模型大量输出 EOS。它用 SCAM mask 控制模型能看哪些历史信息、不能看哪些重复 caption。图里有一个眼睛被划掉的图标，表示某些重复 narration 会被 mask 掉，防止模型直接复制答案；下面正常眼睛表示模型仍然能看有用的历史上下文。
这里的意思是：
LiveStar 训练模型学的是 streaming video-language alignment，而不是 EOS classification。
也就是说，它还是让模型保持和预训练一致的能力：看到图像/视频，就生成有意义的语言描述。
推理时，LiveStar 用 SVeD。图下面有 “Check / Single Forward”。意思是每来一个新帧，它先不急着生成新句子，而是检查上一句 narration 还能不能解释当前画面。
如果还能解释，就输出 Silent。如果不能解释，就 Decode 新 narration。
(d) Training Perspective 是在解释 SCAM 的训练视角。
图里有 image token 和 narration token 交错排列。不同颜色的 narration 代表不同 semantic clip 的 caption。
关键点是：模型训练时不是简单的：
image → captionimage → EOSimage → EOSimage → caption
而是把视频流组织成一个连续上下文，让模型学：
当前帧 + 历史上下文 → 当前语义片段的 narration
同时，SCAM 会 mask 掉容易造成复制泄漏的 caption，只保留必要的历史总结信息。
(e) Full Narration 展示最终效果。
模型输出的不是零散的单帧 caption，而是一段连贯 narration：
一开始，小猫在草地上跑；随后，另一只小猫进入画面；最后场景切换到室内。
不同颜色对应不同时间点输出的 narration。它想说明 LiveStar 的输出是：
少量、及时、连贯、有上下文的 narration。
不是每一帧都说，也不是只会沉默，而是在语义变化时更新描述。
之前的 online video benchmark 有两个问题：
第一，任务太窄。很多 benchmark 主要还是 streaming video QA，也就是边看边回答问题，但真实在线助手不只做 QA，还要做实时解说、实时定位、答案更新、多轮交互等任务。论文也明确说，已有 SVBench、OVO-Bench、StreamBench 等工作虽然开始做 online evaluation，但主要集中在 video question answering，缺少多任务覆盖。
第二，场景太窄。很多 online Video-LLM 主要在 Ego4D 这种第一人称视频上测，但真实场景可能包括直播、监控、宠物、体育、教学、游戏、新闻、娱乐等。OmniStar 因此想做一个更接近真实公开视频平台的 benchmark。
所以 OmniStar 的核心 motivation 是：
以前 benchmark 主要问“模型能不能回答视频问题”，OmniStar 想问“模型能不能在真实视频流中，决定什么时候说、说什么、是否保持沉默，并且保持上下文连贯”。
OmniStar 是一个覆盖 15 diverse real-world scenarios 和 5 evaluation tasks 的 online video understanding 数据集。
15 个场景大概包括：Travel & Events、Sports、Pets & Animals、Music、Autos & Vehicles、Film & Animation、Science & Technology、Education、Howto & Style、News & Politics、Entertainment、Comedy、People & Blogs、Gaming 等。PDF 中也说这些场景进一步按 YouTube 原生标注系统细分成 46 个类别。
任务 1：RNG，Real-time Narration Generation
这是最核心的任务。模型要像直播解说员一样，边看视频边生成 narration。
重点不是每一帧都说，而是：
语义变化时输出，语义没变时沉默。
比如视频里一个人先拿出杯子，然后换成书，再换成手机，模型应该只在关键变化点更新描述，而不是每秒重复一句话。
这个任务主要考察：
response timing + narration quality + narrative coherence。
任务 2：OTG，Online Temporal Grounding
这个任务是给模型一个文本 query，让模型在视频播放过程中实时判断：
这个事件什么时候开始？什么时候结束？
比如 query 是：
A man reporting from outside of a destroyed house.
模型要在视频流中判断相关片段的 start/end，而不能提前看未来帧。
这个任务主要考察：
在线时间定位能力。
传统 temporal grounding 通常是看完整视频后找片段，但 OTG 要求模型在 streaming 设置下做定位。
任务 3：FDQ，Frame-level Dense Question Answering
这个任务有一个固定问题，但答案会随着视频变化而变化。
比如问题是：
What is the person showing to the camera?
答案可能随时间变成：
nothing → white cup → grey cup → red book
模型不能一直重复回答，也不能漏掉变化。它要在答案发生语义变化时更新。
这个任务主要考察：
模型能不能持续跟踪状态变化，并在答案变化时及时更新。
任务 4：COQ，Contextual Online Question Answering
这个任务是在线上下文问答。用户会在不同时间点提问，模型只能基于当前已经看到的视频和历史对话回答，不能看未来。
关键点是后面的问题可能依赖前面的回答。
比如：
Q1：Who is the competitor?A1：A man in red.Q2：What did he continue to do?
这里 Q2 的 “he” 需要模型记住前文指的是谁。
这个任务主要考察：
历史记忆、指代消解、上下文理解。
任务 5：MIQ，Multi-turn Interactive Question Answering
MIQ 比 COQ 更复杂。它不只是一个 QA chain，而是可能同时有多个问题链并行出现。
比如同一个时间点问：
Q1：What appears at the beginning?Q2：What is above it?Q3：What is in front of it?
后面又继续追问。模型要同时维护多个交互线程。
这个任务主要考察：
多轮交互、多问题链跟踪、实时上下文管理。
4. OmniStar 的标注方式有什么特别？
OmniStar 不是简单给每个视频一个 caption 或一个 QA。它做的是 temporally dense annotation。
也就是每个视频被切成多个带时间戳的语义片段，每个片段都有对应描述，而且这些描述不是孤立的，而是构成一条连续故事线。论文举的风格类似：
First, the woman gathers the ingredients...Then, she mixes them in a large bowl...Finally, she places the dish in the oven...
也就是说，它标的不只是“这一帧有什么”，而是“视频故事如何随时间推进”。PDF 里明确说 OmniStar 的 annotations 是 narrative-consistent caption sequences，目的是帮助模型学习 long-range narrative flow。
这点很重要，因为 online video assistant 不能只会输出碎片 caption，它要知道前面发生了什么，然后让后面的输出接上去。
5. 它怎么评测模型？
OmniStar 的评测不只是看答案对不对，还看 输出时间对不对。
主要指标包括：
TimDiff：模型输出时间和 ground truth 时间的差距，越小越好。TimRedun：模型多余输出的次数，越小越好。TimCover：模型覆盖了多少应该响应的场景，越高越好。SemCor：semantic correctness，用 GPT-4o 从语义准确性、语言质量、信息完整性等角度评分。SumFluen：summary fluency，把模型多段输出拼起来，看整体叙事是否连贯。PDF 里对这些指标有详细定义。
这个评测设计的意义是：传统 benchmark 可能只问 “answer right or wrong”，但 OmniStar 会问：
你是不是在正确时间说了正确内容？有没有乱说？有没有重复？前后是否连贯？
这更符合 online assistant 的真实需求。
6. OmniStar 和 SVBench / OVO-Bench 的区别
你可以这样理解：
SVBench / OVO-Bench 更像是在测 online video QA 能力；OmniStar 更像是在测完整的 online video assistant 能力。
区别主要有三点：
第一，OmniStar 不只测 QA，还测 narration、temporal grounding、dense QA、contextual QA、多轮交互 QA。
第二，OmniStar 强调 response-silence timing，也就是模型要自己决定什么时候输出，而不是只在固定问题点回答。
第三，OmniStar 场景更宽，从公开视频平台和真实场景出发，而不是集中在第一人称或单一任务类型上。

模型不是每来一帧都重新生成一句话，也不是输出 EOS 表示沉默，而是检查上一句 caption 在当前新画面下还适不适合。具体流程是：视频帧一帧一帧进来。 第一个语义片段，比如 Frame 1 到 Frame 5，模型生成 Caption 1。 后面新帧继续进入时，模型先不直接 decode 新 caption，而是用一次 forward 计算：当前画面下，旧 Caption 1 的 PPL 有没有明显变高。 如果 PPL 没有明显变高，说明旧 caption 还能解释当前画面，所以输出 Silent。 如果 PPL 超过阈值，说明画面语义已经变了，旧 caption 不再合适，于是触发 decoder，生成 Caption 2。 后面继续重复这个过程。 论文摘要里也明确说，LiveStar 的 response-silence decoding 是通过 single forward pass verification 来决定最优主动响应时机，而不是依赖 EOS token。
训练阶段的 attention mask matrix：横轴和纵轴都是交错排列的：Img, Cap, Img, Cap, Img, Cap ...训练时，作者把视频帧和对应 caption 交错放进同一个序列里，让模型学习 streaming video-language alignment。但这里有一个问题：同一个 semantic clip 里面，连续几帧的 caption 语义可能差不多。如果模型在生成当前 caption 时能看到前面几乎一样的 caption，它就会直接复制，不是真的理解画面。所以 SCAM 要做 mask。蓝色部分：前一个语义片段的信息。 绿色部分：当前或后一个语义片段的信息。 灰色格子：被禁止 attend，防止信息泄漏。 浅色格子：允许 attend。 上面的箭头表示：模型可以保留每个语义片段最后的 caption 作为历史总结，但不能随便看同一片段里重复的 caption。

Online Video Understanding: OVBench and VideoChat-Online CVPR 2025online video understanding 不是简单长视频理解，而是 time-dependent understanding。模型既要记住过去，又不能无限存过去；既要看清当前细节，又要保留长期时间信息。
OVBench 是一个面向在线视频流的 benchmark。它的核心不是让模型看完整段视频后回答，而是在某个当前时间点提问，测试模型能否区分过去、当前和未来。任务上，它覆盖空间感知、时空轨迹、时间感知、过去记忆、时间幻觉验证和未来预测六类能力。构造上，它从多种视频场景中收集数据，利用动作时间、物体位置、轨迹和 timestamp 等高质量时空标注生成 QA，再通过相似物体、错位时间戳等方式构造干扰选项，并人工检查质量。因此 OVBench 更适合评估模型在真实 streaming video 场景下的实时理解能力。

把在线视频理解分成三个时间上下文：Past、Current、Future。基于这三个时间段，它设计了 6 类核心任务：
Spatial Perception：看清当前画面里的空间关系，比如某个人/物体在什么位置。
Spatial-Temporal Perception：理解过去一段时间里的运动轨迹，比如两个人是靠近还是远离。
Temporal Perception：理解动作或物体状态的时间范围，比如某个人什么时候出现/消失。
Hallucination Verification：判断过去看到的东西现在是否还存在，防止模型把历史信息误当成当前事实。
Past Memory：回忆过去某个时间点的状态，比如 13 秒前某个物体在哪里。
Future Prediction：根据当前和过去信息预测接下来会发生什么，比如某个人下一秒会往哪个方向移动。
所以第一张图的核心意思是：
OVBench 不只是问“视频里有什么”，而是测试模型能不能在流式视频中区分过去、现在、未来，并完成感知、记忆、验证和预测。
OVBench 怎么构造。它的生成流程分成四步：
Task formulation：先定义 online video 里的任务类型，也就是 Past / Current / Future 下的不同问题。
Data collection：从多个场景收集视频，包括 movie、instructional、outdoor、indoor、road scenes、open-domain，保证场景多样。
QA construction：利用已有的高质量时空标注，比如动作时间段、物体位置、timestamp、bounding box，再由人工按模板改写成问题。
Option generation：生成选择题选项，并加入更贴近流式视频的干扰项，比如相似物体、错位时间戳、常见错误回答；最后再人工检查问题清晰度、选项歧义和标注准确性。

Dynamic Memory
用阈值 ε 做三种决策。论文默认 ε=0.7。第一种，如果 δ > ε，说明当前帧和上一帧/最后 memory 很像，比如用户手还在同一个位置，画面变化不大。这时候不追加新 memory，而是把当前帧融合进最后一个 memory。直白说，就是“这帧没必要单独存，更新一下最近记忆就行”。第二种，如果 δ ≤ ε 但 σ > ε，说明当前帧和最近画面不像，但和很久以前的某个 memory 有关系，比如同一个物品再次出现、同一个动作再次发生。这时候模型会用 attention 去软更新历史 memory slots，而不是简单丢掉或盲目追加。直白说，就是“这不是最近重复，但它和过去某件事有关，要更新那段长期记忆”。第三种，如果 δ ≤ ε 且 σ ≤ ε，说明这是一个新信息，那就直接 append 到 memory。直白说，就是“这个东西以前没见过，值得新开一个记忆槽”。这个设计的意义是：memory 不是无限增长，也不是强行固定长度；它会对重复内容做融合，对新内容做扩展。 这比纯 token accumulation 更适合 ultra-long video，也比固定 memory bank 更灵活，因为固定 bank 容易在长视频里把早期但重要的信息挤掉。解决：长期第一视角视频怎么存？Eg. Module 1：Streaming video chunk
输入是 AR 眼镜连续看到的第一视角视频流。我们不会一次性拿到完整视频，而是每隔 5 到 10 秒切成一个 chunk。这样更符合真实 AR streaming 场景，也符合 fixed memory budget 的约束。
这里强调两点：不是 offline full video，是 streaming chunk-by-chunk input
Module 2：Video-LLM encoder
每个 video chunk 会先经过 frozen Video-LLM 或 vision encoder，比如 VideoLLaMA3 / Qwen2.5-VL。这里 backbone 不作为我的主要贡献，我主要利用它提取两类信息：一个是 chunk-level embedding，另一个是 visual KV cache。
这里重点是：backbone frozen我的贡献不在训练 base model，我的贡献在 memory manager
第一版可以用开源 Video-LLM 作为 backbone，先冻结它，只训练 memory controller。
Module 3：chunk embedding + visual KV
对每个 chunk，我拿到两种表示。第一种是 chunk embedding，用来判断这个片段属于什么 routine；第二种是 visual KV cache，用来作为真正要保存的细粒度视觉记忆。
换句话说，embedding 负责“路由和管理”，KV 负责“保留可回答问题的视觉证据”。
embedding 用来决定这个片段写到哪里；KV 用来真正回答问题。
Module 4：Learnable Routine Router
Routine Router 的作用是判断当前 chunk 应该写入哪个 routine memory。比如它看到药瓶、水杯、早晨时间、洗手间/厨房场景，可能会把这个 chunk 路由到 medicine routine；看到锅、灶台、食物，可能路由到 cooking routine。
我不想只做 heuristic clustering，而是希望做成 learnable router。它输入 chunk embedding 和已有 routine states，输出这个 chunk 属于每个 routine 的概率。
可以写成：π(r | z_t, h_1, ..., h_K)
z_t 是当前 chunk embedding，h_r 是第 r 个 routine 的状态。router 输出一个分布，决定当前 chunk 更像哪个 routine。
这里的 novelty 小点是：routine 不是人工定义的标签，而是模型在长期视频里学出来的 latent memory state。
Module 5：write KV into M_r
M_r 表示第 r 个 routine 的 KV memory shard。比如 M_medicine 存吃药相关的 KV，M_cooking 存做饭相关的 KV，M_work 存工作相关的 KV。
所以 M_r 是routine-specific 的 layer-wise visual KV cache 子库。M_r = {K_r^1, V_r^1, ..., K_r^L, V_r^L} 因为 Transformer 每一层都有 KV cache，所以严格来说每个 M_r 都是 layer-wise KV shard。
这里你要强调 novelty：
现有方法通常只有一个 flat cache；我这里把 cache 按 routine 拆成多个 shard。
Module 6：update routine state h_r
每个 routine 不只是一个 KV 容器，它还有一个 routine state，记作 h_r。这个 state 是这个 routine 的长期画像，记录这个 routine 通常长什么样、什么时候发生、和哪些 object/action 相关。
例子：
h_medicine: 药瓶、水杯、早晨/晚上、拿药动作h_cooking: 厨房、锅、食物、切菜/加热动作h_work: 桌面、电脑、文档、会议对话
每当新 chunk 被写入 M_r，对应的 h_r 也会更新。后面的 budget allocator 和 query selector 都会用到这个 h_r。
这里强调：h_r 是 routine 的 summary state；M_r 是 routine 的 KV memory。
Module 7：Budget Allocator gives B_r
因为总 KV budget 是固定的，不可能每个 routine 都无限存。所以 Budget Allocator 要决定每个 routine 分多少 memory budget，记作 B_r。
最简单的 baseline 可以是按 frequency 分，比如高频 routine 多给一点 budget。但我的主方法不是手写 B_r = frequency，而是 learnable allocator。
B_r = B_total · softmax(MLP(h_r, stats_r, global_state))
h_r 是 routine state，stats_r 可以包括 frequency、recency、uncertainty、当前 memory size，global_state 表示整体 memory pressure。模型自己学习哪个 routine 更值得存，而不是我手写规则。
frequency 只是输入特征，不是最终规则。
Module 8：Retention Scorer keeps top-B_r KV tokens
Budget Allocator 只决定每个 routine 有多少预算，但每个 routine 内部还要决定具体保留哪些 KV token。Retention Scorer 就是做这个的。
s_i = f(token_i, h_r, timestamp_i, layer_i)
它给 M_r 里面每个 token 打分，然后保留 top-B_r 个 KV token。
重点在 routine-conditioned：
token 的重要性不是绝对的，而是和 routine 有关。同一个药瓶 token，在 medicine routine 里可能非常重要，但在普通厨房背景里可能没那么重要。所以 retention scorer 要 condition on routine state h_r。同一个视觉 token，在不同 routine 里有不同价值。
InfiniPot-V 用 temporal redundancy 和 value norm 做 training-free 压缩，StreamMem 用 generic query tokens 的 attention score 来维护 fixed-size KV memory
Module 9：query selects top-k routine shards
当用户提问时，系统不再把 query 和整个 flat cache 匹配，而是先选择相关 routine shard。比如用户问“我今天吃药了吗”，query selector 应该优先选择 M_medicine、可能再选择 morning/kitchen 相关 shard，而不是扫描所有历史 KV。
score(q, h_r) → top-k M_r
q 是用户问题，h_r 是 routine state。模型选择最相关的 top-k routine memory shards。
StreamKV 是 query 检索 single-video semantic segment；我这里是 query 检索 cross-day persistent routine shard。StreamKV 的 segment 是当前视频里的语义片段，而 RoutineMem 的 routine 是多天、多次出现的行为状态。
Module 10：Video-LLM answers
Video-LLM 只基于 selected routine KV shards 和当前 query 生成答案。这样可以在 fixed memory budget 下避免把所有历史都放进上下文，同时优先保留和读取用户长期生活里最相关的 routine memory。
整个 controller 学的是四件事：where to write, how much to keep, what to evict, and where to read.

双轨架构（routine tree + novelty track）、树形结构、scene 和 time 双信号匹配、episodic 和 semantic 分层。这页是完整版。
流程是这样：每 10 秒切一个 chunk，提取 visual embedding 和 time-of-day feature。阶段一用 Day 1 做 warm-up DP-means 聚类，初始化 ROOT 下面的 top-level routine。阶段二 Day 2-7 是 streaming，每个新 chunk 自顶向下匹配——
在 ROOT 层算 P(child|chunk)，如果 max P 大于阈值并且 margin 足够大，就进入对应顶层 routine；如果都不匹配，进 novelty track。只有 ROOT 层失败才算 novelty，这是"全新大类活动"。
进入 routine 之后再做中间层匹配——如果该 routine 已经有 sub-children，继续向下；中间层匹配不到的话，新建一个 provisional sub-cluster，等重复出现 ≥ M 次再升为正式 child。这样树不会被偶然噪声撑爆。
每个 routine 节点存四样东西：visual prototype、time distribution、固定 K 个 semantic prototype tokens、以及 episodic instances。
Novelty pool 每 N 天扫描一次，里面如果出现 ≥ M 个相似事件——比如连续几个周末去美术馆——就 promote 到 routine tree 里挂在合适位置。
Budget 是固定的，B_total = B_routine + B_novelty 默认 7:3。Query 来时所有 routine 节点和 novelty events 在同一打分空间竞争，beam search 取 top-r。

Active ≠ Relevant: Source-aware Multi-stream VideoScenario：场景 A：多人会议 / presentation场景 B：找东西 / 物体记忆场景 C：实验室 / 医疗 / 安全操作真实 AR / multi-user assistant 面对的是多个 memory sources：user ego view other users’ ego views room/static cameras audio/transcripts 但一个 query 只需要少量 evidence。ChallengeSingle-view may miss evidence.All-view fusion wastes context budget, adds noise, and raises privacy/permission cost.Research questionGiven a query and long-form multi-stream videos, how can we select top-k event-relevant sources/segments under a fixed context budget?
[NOTES]
未来 AR 助手会运行在多人共享空间里，同一个事件可能同时被多个用户的第一视角眼镜和房间摄像头记录。但对某个具体问题来说，只有少数 active streams 真的有用。由于长时多路视频不可能全部放进 VLM context，系统必须在固定 memory/context budget 下判断该查哪一路、哪一段、忽略哪些无关视角。我的问题就是 source-aware / memory-efficient multi-stream video QA：从多路同步长视频中选择并压缩事件相关 evidence 来回答问题。
第一是 计算和 context 成本。CASTLE 这种设定里，15 路视频、600+ 小时内容，本质上是 dataset-scale memory。即使每路只抽低 fps frame，乘以 15 路和长时间窗口后，token 数也会爆炸。你的脚本也一直强调不下载完整视频、不用 hf_hub_download 下载 video，而是只抽指定 clock-time 的少量帧，这说明你的实验已经在围绕低带宽、低存储、低上下文的现实约束做 feasibility。
第二是 无关信息会干扰模型。不是说 context 越多越好。如果把 Kitchen、Reading、Klaus 这种 active 但无关的视角也放进去，模型不但浪费 token，还可能被错误场景误导。比如用户问 living-room presentation，结果 context 里混进厨房、空房间、校准视角，VLM/LLM 可能产生错误关联。
第三是 单视角又不够。如果只看一个 ego view，可能看不到全局事件。AR 眼镜很容易被遮挡，用户也不一定一直看向关键信息。静态摄像头能提供 room-level context，其他 ego views 能提供互补细节。所以你的目标不是 “single best view”，而是 small set of complementary evidence views。
第四是 隐私和权限。真实 multi-user AR assistant 不能随便把所有人的视频都读一遍。更合理的是：只在有必要时访问最相关的 source，并能说明 evidence 来自哪个 stream。也就是 source-aware retrieval / source provenance。
场景 A：多人会议 / presentation
几个人在客厅或会议室讨论一个项目。用户之后问：
“刚才那个人指的是屏幕上的哪一部分？”
用户自己的 AR 眼镜可能只看到桌子或旁边的人；另一个人的 ego view 可能正对屏幕；Meeting static camera 可能看到整个人群和屏幕；Kitchen camera 虽然也在录，但完全无关。
这正好对应你现在的 CASTLE window：Onanong / Tien 是相关 ego views，Meeting static 更直接看到 group event，Kitchen / Reading / Klaus 是 active 但无关。你的研究问题就是：系统如何自动做出这种 source selection，而不是靠人工看 contact sheet。
场景 B：找东西 / 物体记忆
用户问：
“我的门卡刚才放哪里了？”“谁拿走了桌上的剪刀？”
用户自己的眼镜不一定拍到物体被移动的瞬间，因为他可能转身了。但另一个人的眼镜或者房间静态摄像头可能正好拍到。这里 “self memory” 不够，需要跨 source 查证。
但你不能一口气看所有人所有视频。系统要先判断：
哪些 camera 在那个时间段覆盖了桌子？哪个 ego view 离目标物体最近？哪些视角只是同时间 active 但不在同一空间？
这就是 source-aware memory retrieval。
场景C：实验室 / 医疗 / 安全操作
用户问：
“刚才是谁碰了样品？”“试剂有没有被放回冷藏柜？”“手术/实验里某个工具最后在哪里？”
这种场景对准确性和证据来源要求更高。系统必须能说：
我是从哪个 camera、哪个时间段看到的。哪些 source 被排除了，因为它们没有覆盖该事件。
这就把你的问题从普通 video QA 提升成：
source-aware evidence selection with provenance under fixed budget。

--- SLIDE 36 ---
1. 和把所有 video 全合起来的区别（避免把所有stream都塞进模型）：能不能用最少的、对 self view 有增量价值的 source，接近 all-active 的 evidence coverage
2. 除了隐私、传输、context，还有没有更有意思的 constraint
现在更核心的 constraint 不是单纯省 token，是 incremental information。其他人的 video 里很多内容和 self view 是重复的，或者完全 off-event。我们真正想找的是：self view 没看到、但其他 source 看到了，并且和 query 相关的 evidence。所以后续自动方法不能只是选最相似的 view，因为最相似可能只是重复；应该选 query-relevant 但和 self view non-redundant 的 view。

--- SLIDE 37 ---
3. 应该先查自己的 video，还是直接查别人？现在把 pipeline 改成 self-first。真实 AR assistant 里，self view 默认权限最低、成本最低、隐私风险最低。所以系统应该先判断 self view 是否 sufficient；如果 sufficient，就不访问别人；如果 insufficient，再查 selected other/static source。结果里 2/6 的问题 self-only 就够，所以系统确实不应该每次都访问别人；但 4/6 的问题 self-only 不够或不确定，所以只看自己的 memory 也不够。4. 别人看到的大部分和自己很像，怎么找自己没看到的？先用人工 oracle 做 proof-of-concept。对每个 QA，手动 audit 哪个 view 相对 self view 提供了 missing evidence。后续自动 selector 可以做成 relevance_to_query - λ * redundancy_with_self（先验证这个现象存在；下一步才做自动 selector）5. 如果不中心化，别人的 video 在 device 上，怎么搜？这次先假设 centralized index，因为现在只想验证 core assumption：other source 是否真的有增量 evidence。如果这个 assumption 成立，下一步可以考虑更系统化的 access control：先访问低成本 summary / metadata，再决定是否请求 raw video。这个可以作为后续 extension。

--- SLIDE 149 ---
MOTIVATION + LINEAGE
From single-view memory QA to two-user evidence QA
Problem: an egocentric assistant should answer questions that depend on what multiple people saw, not just one wearer’s stream.
Why two users?
Many real daily-life questions are distributed: one person observes an object, another observes the destination, hand-off, or later state. Any single view is incomplete.
EgoEverything inspiration
Borrow the evidence-first habit: select evidence, generate an MCQ around supported facts, then use a reviewer loop to reject weak or ambiguous questions.
EgoLife adaptation
Replace single-video selected-object QA with multi-user semantic complementarity: each required user contributes a distinct fact to the final answer.
single-view QA
evidence-first generation
two-user answerability
the QA item should be answerable only after combining at least two users’ evidence slices.
Key distinction: inspired by EgoEverything’s QA construction loop, but the target unit is a multi-user EgoLife event, not a single selected object in one clip.

StreamForest
[2509.24871] StreamForest: Efficient Online Video Understanding with Persistent Event Memory
NeurIPS 2025 Spotlight
[NOTES]
Streamforest 的核心思想是长期记忆按事件树进行存储，即把 Memory 模块分成一个个 MetaEvent 节点，再将多个事件节点组织成一片 Forest。每棵树代表经过层次化合并后的事件记忆结构。
接下来具体讲一下这篇论文的方法：
1. Fine-Grained Spatial-Temporal Window
(a) Real-time Perception：直接保留当前帧的视觉 Token，并在特征中加入时空位置信息。
我觉得这也是一个我们可以参考的点：对于每个事件，加入它的时间信息和空间信息。这样就可以便于大模型做出一些规律的识别，例如可以检测：  1. 前后两次同样事件间隔的时间 2. 在一天、一周或者一个月内，是否存在固定的发生时间（比如都是在周末，或者都是在早上）
(b) 短期时空记忆：新帧到来后，旧帧不会立刻丢弃，而是先存入一个持续 TS 秒的短期记忆序列。
(c) 模型会计算窗口内新旧帧之间的相似度，为后续事件切分做准备。
2. 长期记忆模块 (PEMF)
当短期记忆满了之后，会送入长期记忆模块（Persistent Event Memory Forest）并进行事件切分。
(a) 切分做法：在连续帧序列中利用帧间相似度寻找局部最小值。局部最小值对应明显的状态变化点，以此将视频划分为若干 MetaEvent。
(b) 记忆维护：系统设有明确的 Vision Token 上限。当 Token 超过上限时，必须执行层次化的记忆整合。
(c) 合并策略：模型在相邻 MetaEvent 节点中寻找最适合合并的一对。合并依据是“总惩罚分数”，由以下三项组成：
- 相似度惩罚：计算两个节点 Token 间的余弦相似度矩阵，选出 TopK 最相似的对，取平均相似度。相似度越高，惩罚越低。
- 合并次数惩罚：防止局部区域被反复压缩。
- 时间距离惩罚：对旧事件的压缩容忍度更高。
然后执行合并，合并的方式使用 ToMe，同时更新这个新的 MetaEvent 的时间戳。
当用户在某一时刻提出问题时，模型会把 PEMF 中所有根节点的长期事件记忆，以及 FSTW 中保存的当前帧和短期时空记忆同时送入 LLM。
这个时候，LLM 就可以同时访问：
1. 当前的局部细节
2. 最近的短期动态
3. 长期记忆中的一些事件级摘要

--- SLIDE 211 ---
Working-Episodic-Semantic
Working Memory 对应当前高分辨率 FSTW/KV cache，负责实时感知和当前 query；Episodic Memory 对应带有时间、地点、人物、物体、动作和证据指针的 EventNode，负责保存具体发生过的事件；Semantic Memory 对应由多个 EventNode consolidation 得到的 PatternNode、RoutineNode 和个人语义事实，负责表示用户长期习惯、周期性行为、稳定物体位置和人物关系。在线阶段，视频流首先进入 Working Memory；当检测到 event boundary、显著性提升、用户 query 或 prediction error 时，系统将短期窗口写入 Episodic Memory。离线或低负载阶段，系统通过 replay、聚类和模式挖掘，将重复的 episodic events 抽象为 semantic patterns，并对冗余事件执行压缩或遗忘。查询阶段，系统根据问题类型动态选择检索层级：当前感知问题主要访问 Working Memory，近期回忆问题访问 Episodic Memory，习惯/规律/异常问题访问 Semantic Memory，并在固定 token/KV budget 下组合少量相关证据，从而支持 lifelong video understanding 中近似 O(C) 的推理成本。
lifelong video memory 不只要理解当前画面，还要能回忆具体历史事件，并抽象出用户长期习惯和规律。
